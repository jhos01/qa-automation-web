## рџ“ќ 4. Convenciones de Versionamiento (Reglas de Oro de Git)

Para demostrar que trabajas de forma limpia y profesional, sigue estas reglas al guardar tus cambios.

### Commits: Mensajes Claros y Descriptivos 

Usa el formato: `tipo(scope): mensaje`

- `tipo`: ВїQuГ© tipo de cambio hiciste? 
  - `feat`: Si agregaste una nueva funcionalidad (ej. una nueva prueba). 
  - `fix`: Si corregiste un error (un bug). 
  - `docs`: Si solo cambiaste la documentaciГіn (como un `README.md`). 
  - `test`: Si agregaste o modificaste pruebas existentes. 
  - `chore`: Tareas de mantenimiento que no afectan el cГіdigo (ej. actualizar el `.gitignore`). 
- `scope` (opcional): ВїQuГ© parte del proyecto modificaste? (ej. `login`, `users-api`, `reports`)
- `mensaje`: Una descripciГіn corta y en minГєsculas de lo que hiciste.

**Ejemplos:** 
- `feat(users): agrega prueba para crear usuario con datos vГЎlidos` 
- `fix(login): corrige el selector para el botГіn de enviar` 
- `docs(readme): actualiza la secciГіn de tecnologГ­as usadas` 
- `test(reports): implementa la generaciГіn de reportes en HTML` 

### Ramas (Branches): OrganizaciГіn del Trabajo

Crea una nueva rama para cada tarea importante. Usa el formato: `stageX/tipo/nombre-descriptivo` 

- `stageX`: El mГіdulo en el que estГЎs trabajando (`stage1`, `stage2`).
- `tipo`: El tipo de entrega (`quicktask`, `challenge`, `mission`). 
- `nombre-descriptivo`: Una descripciГіn corta separada por guiones. 

**Ejemplos:** 
- `stage1/challenge/automatizar-crud-usuarios`
